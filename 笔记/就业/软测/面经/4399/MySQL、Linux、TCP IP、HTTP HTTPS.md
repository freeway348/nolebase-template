## 一、MySQL 高频考点

### 1. 存储引擎：InnoDB vs MyISAM

面试官问"用过什么数据库"后，通常会追问存储引擎。如果你被问到“MySQL的储存索引引擎”，回答**MyISAM、InnoDB**两种最常用的存储引擎即可（很多面经把"存储引擎"问成了"索引引擎"，实际是同一个问题）。

**✅ 建议回答：**
> 最常用的两种存储引擎是 **InnoDB** 和 **MyISAM**。InnoDB 支持**事务**（提交、回滚、崩溃恢复），是现在的默认引擎；MyISAM 不支持事务，但读速度快。InnoDB 支持**行级锁**和**外键**，MyISAM 只支持表锁。实际项目中我都用 InnoDB，因为它的事务特性保证数据一致性。

> **💡 加分话术**（如果你用了 InnoDB）："我在项目里主要用 InnoDB，因为需要保证数据一致性。比如测试‘安享智慧理财’的借款还款接口时，涉及金额的增减必须放在一个事务里，要么都成功要么都回滚。"

### 2. 索引：B+树与最左前缀原则

**（1）MySQL 为什么用 B+ 树？**

**✅ 建议回答：**
> MySQL 默认用 **B+ 树**作为索引结构，主要两个原因：一是 B+ 树所有数据都在**叶子节点**，且叶子节点用**链表连接**，做范围查询（比如 `WHERE id > 100`）时直接遍历链表，效率很高；二是**非叶子节点只存索引不存数据**，一个节点能放更多索引键，树的高度更低，磁盘 IO 更少。

> **💡 加分对比**：如果面试官追问"为什么不用哈希索引"，可以说"哈希索引等值查询是 O(1) 很快，但不支持范围查询和排序，所以 InnoDB 默认用 B+ 树更通用。"

**（2）最左前缀原则（必考！）**

如果你有联合索引 `(a, b, c)`，查询必须从最左列开始匹配。

**✅ 建议回答：**
> `WHERE a=1 AND b=2` 会走索引，`WHERE b=2` 不走索引。优化建议是把高频查询的字段放在联合索引的最左侧。

**（3）索引失效的常见情况**

**✅ 建议回答：** 直接说出这几种场景即可：
> 1. 查询条件用 `OR` 连接；2. `LIKE '%xxx'` 以通配符开头；3. 对索引列做计算或函数操作；4. 违反最左前缀原则。


### 3. 事务 ACID（必考）

**✅ 建议回答：**
> **A 原子性**：事务要么全执行，要么全回滚（通过 Undo Log 实现）；**C 一致性**：数据必须符合业务规则，比如转账后总金额不变；**I 隔离性**：多个事务互不干扰（通过锁和 MVCC 实现）；**D 持久性**：事务提交后数据不丢失（通过 Redo Log 实现）。

### 4. SQL 查询：JOIN 与慢查询优化

**（1）内连接（INNER JOIN）**

**✅ 建议回答：**
> INNER JOIN 返回两个表中**匹配的数据**。比如查玩家的充值记录，用 INNER JOIN 连接玩家表和充值表，只返回有充值记录的玩家。

**（2）慢查询优化步骤**

**✅ 建议回答：**
> 第一步：开启 `slow_query_log` 定位慢 SQL；第二步：用 `EXPLAIN` 看执行计划，重点关注 type（访问类型，尽量避开 ALL）、key（是否用到索引）、rows（扫描行数）；第三步：优化——加索引、避免 `SELECT *`、子查询改 JOIN。


### 5. 数据库场景题

面试官可能结合你简历中的项目发问。

**✅ 建议回答（结合你的项目）：**
> 在"安享智慧理财"项目中，我测试借款和还款接口时，会直接查数据库验证**数据的正确性**。比如测试还款功能，我会用 SQL 查还款记录表，看金额、状态、时间是否与预期一致；也会用多表联查（JOIN）关联借款表和还款表，验证还款总额是否超过借款金额。如果发现数据不对，就说明接口逻辑有 bug。

## 二、Linux 高频考点

测试工程师被问 Linux，核心是考察你能否**独立定位服务端问题**——面试官想知道你不是只会"点按钮"，而是能登录服务器排查日志。

### 1. 查日志：grep（90%必问）+ tail（85%必问）

*   **`grep`**：日志搜索神器。搜索报错信息（`grep "ERROR" app.log`）、统计出现次数（加 `wc -l`）、查看上下文（`-C 5`）。
*   **`tail`**：查看最新日志。`tail -f` 实时监控（新功能部署验证）、`tail -100` 查最后100行（分析批量任务失败）。

**✅ 建议这样回答：**
> 我主要用来**查日志定位问题**。最常用 `grep` 和 `tail -f`。比如压测时发现接口返回 500 错误，我会登录服务器用 `grep "ERROR" /var/log/app.log` 搜错误日志定位具体报错。部署新版本后，用 `tail -f app.log | grep "ERROR"` 实时监控日志。另外常用 `ps -ef | grep java` 确认服务是否启动，`top` 看 CPU 和内存使用情况。

### 2. 其他常考命令速记

| 命令 | 用途 |
|:---|:---|
| `ls -l` / `ll` | 查看目录文件详情 |
| `cd` / `cp` / `mv` / `rm` | 切换/复制/移动/删除 |
| `chmod` | 改变文件权限（r=4, w=2, x=1） |
| `ps -ef \| grep java` | 查 Java 进程是否启动 |
| `top` | 动态查看 CPU 和内存，看 `load` 值判断负载 |
| `kill -9 PID` | 强杀进程 |
| `df -h` | 查看磁盘使用情况 |
| `free -m` | 查看内存使用情况 |
| `cat / more / less` | 查看文件内容 |
| `netstat -nltp` | 查看端口被哪个进程占用 |
| `tar -zxvf` / `unzip` | 解压文件 |
| `find / -name "*.log"` | 按文件名查找文件 |


### 3. 实战技巧：把简历中的技能变成具体场景

**✅ 面试可以这样串起来：**
> "简历里我写了熟悉 Linux 命令，在项目里主要用它们做三件事：
> 1. **部署验证**：新版本上线后，先用 `tail -f` 实时监控日志，再用 `ps -ef | grep` 确认服务正常启动；
> 2. **Bug 定位**：压测发现接口报 500，用 `grep` 搜异常关键词，定位到后端代码的空指针问题；
> 3. **环境排查**：怀疑端口冲突时用 `netstat -nltp` 确认，磁盘满了用 `df -h` 清理日志。"

## 三、TCP 与 UDP（必考）

### 1. TCP 与 UDP 的核心区别

这是4399及所有软件测试岗的网络协议必考题。

**✅ 建议回答：**
> TCP 像**打电话**——先拨通再说话，保证对方一定听到，可靠但慢；UDP 像**发短信**——发出去就不管对方收没收到，不可靠但快。

| 对比项 | TCP | UDP |
|:---|:---|:---|
| 连接 | **面向连接**（三次握手） | **无连接** |
| 可靠性 | 可靠，保证数据按序到达 | 不可靠，可能丢包、乱序 |
| 速度 | 慢（需维护连接状态） | 快（无需建立连接） |
| 传输方式 | 面向**字节流** | 面向**报文** |
| 适用场景 | HTTP/HTTPS、文件传输、邮件 | DNS、视频直播、**在线游戏** |
| 头部开销 | 20~60 字节 | 仅 8 字节 |

### 2. 三次握手（必考）

**✅ 建议这样回答（用生活化比喻）：**
> 就像两个人打电话确认对方能听到：A 说"喂，能听到吗？"（第一次握手）；B 说"能听到，你能听到我吗？"（第二次握手）；A 说"我能听到，我们开始聊吧"（第三次握手），然后开始通话。

> **❗面试官很可能追问"为什么不是两次握手？"**：因为网络可能延迟。如果只用两次握手，客户端发了一个连接请求 A，结果 A 因为网络拥堵在路上延迟了很久。客户端等不及，又发了一个新请求 B，服务器响应 B，连接建立、通话结束。这时候延迟的 A 终于到了服务器，服务器以为客户端又想建立连接，就傻傻等着，白白浪费资源。

### 3. 四次挥手（必考）

**✅ 建议这样回答：**
> 四次挥手是双方各主动关闭一次连接的过程。打个比方：A 说"我说完了，先挂电话"（第一次，A 不再发送数据）；B 说"好的，我知道了"（第二次，但 B 可能还有话没说完）；等 B 把话说完后，B 说"我也说完了，可以挂了"（第三次）；A 说"好的，再见"（第四次）。等待 2MSL 时间后正式断开，确保最后那条 ACK 对方收到了。

> **❗面试官可能追问"为什么是四次，三次不行？"**：因为 TCP 是双工的，关闭连接前，B 可能还有数据没发完。如果只用三次挥手（合并第二和第三次），B 剩下没发完的数据就丢了。

### 4. UDP 与游戏行业

UDP 在游戏中的关键作用——这也是4399面试可能追问的[游戏专属]视角。

**✅ 面试可以主动聊这个亮点：**
> 在线游戏中，**实时操作**（如英雄移动、技能释放、射击命中）通常走 UDP 协议，因为这类数据传输追求低延迟，丢几个包可以接受，但卡顿会让玩家体验极差。而**登录、充值、交易**等操作必须用 TCP，因为涉及数据可靠性和安全性。这正是游戏测试需要特别关注的：测试实时对战时，要模拟弱网环境看 UDP 丢包对游戏体验的影响。这也与我用 Postman 测接口的经验有相通之处——都是验证数据传输的正确性和稳定性。

## 四、HTTP 与 HTTPS（必考）

### 1. HTTP 与 HTTPS 的核心区别

**✅ 建议一针见血地说出差异：**
> 核心差异是"是否加密"：HTTP 明码传输，HTTPS 加密传输。端口也不同（80 vs 443），HTTPS 需要 CA 数字证书。

**✅ 回答思路与完整话术：**

> 两者的核心差异在于**安全性**。HTTP 是明文传输，数据在互联网上流转就像明信片，任何人拿到都能看；HTTPS 是在 HTTP 和 TCP 之间加了一层 **SSL/TLS 加密层**，对数据进行加密传输。
>
> 具体区别：第一，端口不同，HTTP 默认 80，HTTPS 默认 443；第二，HTTPS 需要从 **CA 机构**申请数字证书来验证服务器身份，HTTP 不需要；第三，HTTPS 是加密传输，HTTP 是明文传输，所以涉及密码、支付的页面，必须用 HTTPS。

> **❗面试官可能会追问"HTTPS 的加密原理"（加分项）** ，你可以这样回答：
> HTTPS 采用**混合加密**：用非对称加密（公钥/私钥）来安全地交换一个"会话密钥"；后续所有数据传输用这个会话密钥进行**对称加密**——因为对称加密速度快，适合大量数据传输。至于证书的作用，证书由 CA 机构签发，包含服务器的公钥，客户端用浏览器内置的根证书来校验它是否合法，防止中间人攻击。

### 2. GET 与 POST 的区别

**✅ 建议回答：**
> GET 用于获取数据，参数拼接在 URL 后面，有长度限制；POST 用于提交数据，参数放在请求体里，更安全。

### 3. OSI 七层模型（你面经中出现了）

**✅ 建议这样先概括出核心对应关系：**
> OSI 七层是网络通信的理论模型，从下到上分别是：物理层（网线/光纤）、数据链路层（MAC 地址）、网络层（IP 地址/路由）、**传输层（TCP/UDP）** 、会话层、表示层、**应用层（HTTP/HTTPS/HTTPS）**。

> **💡 加分亮点**：可以补充一句——"实际工作中我们更常用 TCP/IP 五层模型，把应用层、表示层、会话层合并为应用层。工作中最常接触的是应用层（用 Postman 测 HTTP/HTTPS 接口）和传输层（排查 TCP 连接的建立与断开）"。
